Overview
Executive Summary

PhaseAlpha is building a SaaS product that automates the extraction of HVAC equipment schedule data from PDF building plan sets. Today, HVAC contractors manually transcribe this data for pricing and configuration — a slow, error-prone process where the contractor owns all risk.

This document defines the technical architecture required to validate that problem in Phase 0. It is grounded in analysis of real engineering plan sets from three firms (Bluestone Engineering, MODUS Engineering, and Shive Hattery), hands-on extraction testing, and research into the relevant tool landscape.

Core finding: this is not just an extraction problem. It is a detection + rendering + extraction + trust problem. A naive single-pass approach will appear to work in demos but produce unreliable data in production. The architecture must be fail-safe, not merely high-accuracy.
Key constraint: bad data is worse than incomplete data. Pricing or estimating off fabricated values creates downstream liability risk for HVAC contractors. The system must return null for unreadable values — never infer or guess.

Strategy
Recommended Pilot Path

Phase 0 — n8n Pilot (Weeks 1–3)

Build and test the full extraction workflow in n8n. This gives the team visual workflow control, fast iteration, and built-in HTTP/API connectors without writing backend code. All seven pipeline stages can be wired up, tested against real PDFs, and tuned before any .NET work begins.

Why n8n first: The pipeline has 7–10 distinct stages with conditional branching, fallback logic, and retry behavior. Wiring this directly into .NET before the extraction logic is proven wastes engineering time. n8n makes the workflow visible and testable in hours, not days.

Document Analysis
What the Sample Documents Revealed

Schedule Complexity Varies Dramatically by Firm

FirmProject TypePDF TypeComplexity
Bluestone EngineeringHospitalVector6 schedule types per page, split by AHU system, ~15 columns each
MODUS EngineeringSchoolImage-based8 schedule types including a Rooftop Unit Schedule spanning full 30"×42" sheet with 40+ columns
Shive HatteryStorm shelterImage-based15 distinct schedule tables on a single page — AHU, Fan, VAV Box, Louver, Diffusers, and more

Single-Pass Extraction Fails on Complex Pages

Testing confirmed that rendering a full schedule page and sending it to a vision model in one pass produces unreliable results for dense or full-width tables. The two-pass approach — detect table regions at page scale, then crop and re-render each table at higher DPI — materially improves extraction quality.

Test result: VAV box schedule confidence improved from 0.76 to 0.91 after cropping. Full data values that were illegible at page scale became fully readable. The two-pass crop pipeline is not optional.

Full-Width Schedules Are a Distinct Problem Class

AHU and Rooftop Unit schedules spanning the full width of a 30"×42" sheet cannot be extracted from a single render at any practical DPI. These require a horizontal crop-and-stitch approach: split the table into left and right segments, extract each independently, and reassemble by matching column positions.

Heuristic: if a detected table spans more than 80% of the page width, trigger the horizontal split pass.

Architecture
Full System Architecture

The system requires seven distinct functional layers. Each layer has a specific job and should not be collapsed with adjacent layers.

1
Document Ingestion & Routing
Handled upstream by PhaseAlpha — Plus1 scope begins at Layer 3

PhaseAlpha's existing system handles plan PDF ingestion, page splitting, and schedule page routing. This pipeline receives individual, pre-split schedule pages as PDF files — not full plan sets.

Input format: PDF is required. Even though PhaseAlpha's system can output either PDF or image format, PDF is the correct choice for this pipeline. Submitting pages as PDF rather than flat image files preserves the ability to control rendering resolution (DPI) at extraction time. The crop orchestration layer (Layer 4) renders pages at 150 DPI for detection, then at 300–500 DPI per table based on density — this precision is only possible when the source is a PDF. A flat image has a fixed resolution baked in at capture, which may be insufficient for dense tables and cannot be upsampled reliably. PDF in, crop-and-render on our side.
  • Plus1 receives individual schedule page PDFs via API payload from PhaseAlpha's system
  • Each page is processed as a discrete job — no splitting, no classification required
  • Source traceability metadata (project ID, page reference) should be included in the payload from PhaseAlpha
  • Pipeline begins immediately at table detection (Layer 3)
2
Page Classification
Not required — PhaseAlpha delivers pre-classified schedule pages

Page classification is handled by PhaseAlpha's upstream system before pages are submitted to this pipeline. Every page received is a confirmed schedule page. The classifier layer is documented here for completeness but is not built or billed as part of this engagement.

Scope note: Removing this layer eliminates the Claude Haiku classifier call (~$0.0003/page) and the associated routing logic from Phase 0 build scope. Pages received are assumed to be schedule pages ready for table detection.
3
Table Detection

Identifies every distinct equipment schedule table on a classified page, returning a bounding box and table type label for each.

Detector A — LlamaParse (Recommended Primary)

Combines traditional text extraction with agentic LLM/VLM reasoning. Handles messy HVAC schedules well — reconstructs reading order and preserves context across pages.

Detector B — Docling (Fallback)

Free, MIT-licensed toolkit using computer vision (DocLayNet + TableFormer). Runs fully locally — no cloud dependency, strong data privacy story.

4
Rendering & Crop Orchestration
Most commonly underbuilt — do not skip

Takes table bounding boxes from the detector and produces correctly-sized, high-DPI cropped images for the extraction model.

5
Extraction

Sends cropped table images to a vision model and receives raw cell-level content. This layer must operate on crops only — never on full-page images.

Model recommendation: Claude Opus/Sonnet 3.5/4 is the recommended primary extraction model. GPT-4o is a valid alternative for comparison testing.

Critical failure mode: a model can produce suspiciously complete rows from an unreadable crop by inferring plausible-looking values. The extraction prompt must explicitly prohibit this. Unreadable = null, always.

6b
Schema Normalization

Once column headers are mapped, normalization slots values into the canonical schema — handling unit conversion, terminology normalization, and remarks concatenation.

OpenAI Structured Outputs is recommended as the primary normalization layer because it enforces the JSON schema at the model response layer, making schema violations impossible.

Product Requirements
Review UI Requirements

The review UI is where human judgment corrects what the AI got wrong. PhaseAlpha's development team owns the Review UI — it is not in scope for this engagement. Plus1's deliverable at the review interface boundary is a clean, fully-flagged JSON payload delivered to a defined API contract. The requirements below document what that payload must support so PhaseAlpha's UI team has a complete specification to build against.

Plus1 deliverable at this boundary: A structured JSON response per extracted table containing normalized field values, per-field confidence scores, failure codes, flagged cells, crop image reference, and source traceability. The contract for this payload will be defined as a required handoff artifact before PhaseAlpha's UI development begins.

Core Requirements

  • Side-by-side view: cropped table image on the left, extracted JSON fields on the right — without the source image, review is slow and unreliable
  • Flagged cells highlighted in the extracted view — reviewers see exactly which fields triggered a flag
  • Inline editing: reviewer edits a field directly in the UI, change saved with reviewer ID and timestamp
  • Row-level approve/reject — granular enough to pass clean rows while blocking uncertain ones
  • Bulk approval for high-confidence tables — reviewers should not touch records above a confidence threshold

Review Queue Routing Rules

FABRICATED_COMPLETENESSMandatory review — extraction blocked from export

Platform & Stack Options

Testing Phase Options

The testing phase uses a two-track approach: start with the fastest viable path and escalate to the full dual-detector stack if single-pass extraction proves insufficient.

Start Here
Option A — Fast Path
Single Detector · LlamaParse + Claude
  • Orchestration: n8n pilot
  • Input: pre-split schedule pages as PDF (provided by PhaseAlpha)
  • Detector: LlamaParse (single pass)
  • Extraction: Claude vision (Sonnet/Opus)
  • Normalization: OpenAI Structured Outputs
  • Validation: rule engine
Fastest path to a working pilot — 1.5 to 2 weeks. All code, workflows, prompts, and data models built here are owned entirely by PhaseAlpha. PhaseAlpha is not locked into any vendor platform or product roadmap. This architecture is built on independent APIs that can be swapped, extended, or migrated at any time.
Escalation Path
Option B — Full Pilot / Dual Detector
Dual Detector · LlamaParse + Docling Fallback
  • Orchestration: n8n pilot
  • Input: pre-split schedule pages as PDF (provided by PhaseAlpha)
  • Detector A: LlamaParse (primary)
  • Detector B: Docling (automatic fallback on low confidence)
  • Extraction: Claude vision (Sonnet/Opus)
  • Normalization: OpenAI Structured Outputs
  • Validation: rule engine + optional cross-model verification
⚠️ Option B may be required to achieve extraction goals. Single-pass extraction (Option A) carries a high risk of failure on complex, image-based, or full-width HVAC schedules — even with well-tuned prompts. If Option A accuracy falls short during testing, escalating to Option B's dual-detector fallback is the planned next step, not a redesign. The code carries forward entirely.

Implementation
Prompt Strategy & Automation Sequence

The 7 Prompt Sets

The system requires seven distinct prompt sets. These should not be collapsed — each has a specific job and different quality requirements.

Prompt 1
Page Classifier
Classifies page type from a full-page render. Output: { page_type, confidence, reason }. Keep cheap and fast — use Haiku or equivalent.
Prompt 2
Table Detector
Identifies all schedule tables on a classified page. Output: array of { table_name, equipment_type, bbox_pct }. Must detect multiple tables per page reliably.
Prompt 3
Raw Extraction
Extracts raw cell values from a cropped table image. Hard rule: unreadable = null. No inference. No completion. This is the anti-hallucination gate.
Prompt 4
Column Header Mapper
Maps variant column headers from raw extraction to canonical field names. Input: array of raw headers. Output: { raw_header: canonical_field } mapping.
Prompt 5
Remarks Consolidator
Combines remarks column text, numbered note fragments, and row-tied callouts into a single remarks_combined string. Must handle footnote references.
Prompt 6
Normalization
Produces normalized JSON matching the canonical schema. Uses Structured Outputs to enforce schema compliance. Never backfills unknowns — preserves null.
Prompt 7
Anomaly Validator
Checks a normalized record for integrity issues: duplicate tags, impossible values, suspicious completeness. Output: array of { failure_code, field, description }.

Automation Sequence

A1
Schedule Page Intake
Trigger: PhaseAlpha sends a pre-split schedule page as a PDF via API payload. Creates job record with source metadata (project ID, page reference) provided in the payload. No splitting or classification required — page is already confirmed as a schedule page.
A2
PDF Pre-check
Trigger: page received. Validates that the payload is a readable PDF (not corrupted or blank). Logs page type (vector vs. image-based) for DPI routing decisions in crop orchestration. Routes immediately to table detection — no classification step needed.
A3
Detector A with Fallback to B
Trigger: schedule page ready. Runs LlamaParse (Option A — Fast Path). If testing escalates to Option B, Docling runs automatically as fallback when LlamaParse confidence falls below threshold. Both results persisted during pilot for comparison.
A4
Crop Orchestration
Trigger: bounding boxes available. Crops each table, re-renders at appropriate DPI, splits full-width tables, retries with expanded crop on weak confidence.
A5
Extraction Pipeline
Trigger: crop ready. Sends crop to vision model. Returns raw cells with null for unreadable values. Stores source-to-output traceability.
A6
Normalization Pipeline
Trigger: raw extraction complete. Maps columns to canonical schema. Normalizes values. Concatenates remarks. Assigns confidence scores.
A7
Validation & Anti-Hallucination
Trigger: normalized record created. Runs business-rule checks, detects suspicious patterns, optional cross-model verification, assigns failure codes.
A8
Review Queue Output
Trigger: low confidence, anomaly found, or detector mismatch. Plus1 delivers a structured JSON payload containing normalized field values, per-field confidence scores, failure codes, flagged cell references, crop image URL, and source traceability. PhaseAlpha's development team owns the Review UI that consumes this payload — the API contract for this handoff is a required deliverable before PhaseAlpha UI development begins.
A9
Export to Takeoff Engine
Trigger: record approved. Writes normalized JSON. Optionally writes Excel/CSV. Attaches to project/job record.
A10
Learning Loop
Trigger: reviewer edits output. Stores original extraction alongside corrected version. Tags failure type. Feeds evaluation set for prompt and rule tuning.

Long-Term Plan
n8n Pilot → .NET Production Migration

The n8n pilot is the right first step — it makes the pipeline visible, testable, and fast to iterate. Once the pilot has validated accuracy targets and finalized the extraction stack, the logical next phase is migrating the proven logic into PhaseAlpha's .NET platform for production integration. This is not a redesign; it is a translation of proven logic into a new runtime.

What carries over intact: All seven prompt sets, the canonical data model, the column header synonym knowledge base, the validation rules and failure codes, the crop orchestration DPI logic, and the API payload contract for the review queue handoff. None of this is n8n-specific — it is extraction logic that lives independently of the orchestration layer.
