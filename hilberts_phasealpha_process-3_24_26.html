Optimized Solution Questions
Don't present these as a pitch — build the future state with them. Ask each question, let them talk, capture their reaction. Their answers define what we build.
Rule: Ask one question, then go quiet. Let them fill the silence. You're validating, not pitching. If they react negatively, write it down — that's the most important data.
Step 1 — Structured Job Setup
1
"What if every job had a structured setup — floors, rooms, equipment — instead of rebuilding that structure each time a floor gets released?"
We know they'd want this. Listen for: relief, skepticism about upfront data entry, questions about who enters it. Tom will want to know if it's more work at the start. Context: Dragonfly was partially set up but Tom dropped out — this has to feel lighter than last time.
2
"When you get a job, is the floor and room structure pretty well defined in the drawings — or does it change a lot between bid time and post-award?"
Critical for data model. If room counts are stable at bid time, we can build a single structured record. If they change significantly, we need versioning. AlphaTakeoff reads this from drawings — confirm they'd trust the extracted output.
3
"Who would actually set up the job structure in the system — Tom, Belinda, or someone else?"
Adoption risk: Tom started Dragonfly as a test and dropped out because he didn't finish the guinea pig phase. If job setup requires Tom, it may stall again. Belinda put a lot of work into the first Dragonfly setup — she may be the right person to own this.
Step 2 — Room Data Once, Reused Everywhere
4
"If you entered room data once — room number, equipment designation, CFM, size — could that same data power both the room schedule and the floor orders automatically?"
This is the core architectural argument. We know the data is re-entered at least twice (takeoff → room schedule → floor orders). If they confirm all three use the same fields, the case for a single structured record is ironclad. Listen for edge cases: products that don't follow the same pattern.
5
"What fields absolutely have to be in that room record to make it useful for ordering?"
This defines the data model — write down every field they name. We know from the call: room number, equipment designation (CG/RG/EF1/VAV/FCU), room size, CFM, static pressure. Ask what else. Products to quote may vary by manufacturer rep relationship.
6
"Does that data ever change between bid time and post-award — or is it usually the same fields, just populated more completely?"
Tom mentioned that room schedules can be 10–20 pages for large jobs. If the structure is stable but just gets more complete after award, we design one model with progressive population. If the structure changes, we need bid-version and award-version as separate records.
Step 3 — Auto Schedule Generation
7
"What would it change for your team if the room schedule built itself the moment you got the PO — instead of someone starting from scratch?"
This is their biggest post-award time sink — 10–20 pages of manual document creation. AlphaTakeoff already does this. Listen for: how much time they think it would save, any concern about accuracy of auto-generated output, whether they've ever tried anything like this before (they haven't).
8
"What format does the engineer need the room schedule in — Word doc, Excel spreadsheet, PDF, or something specific to the project?"
AlphaTakeoff outputs editable spreadsheets. If engineers need a specific Word template or a particular column layout, we need to know before we design the output. This is a blocker if they haven't confirmed it.
9
"Once the room schedule goes to the engineer, does it come back with changes — or does it usually go through clean?"
Determines if we need an editable output (Excel) vs. a locked PDF. Also tells us how often there's a revision loop and whether that's a separate pain point worth addressing.
Step 4 — Floor-Based Order Automation
10
"If a contractor emails that Floor 2 is released, should the system generate that floor's order pre-populated and show it to you to review before it goes anywhere?"
Tom explicitly said: "In the middle of this process, there is human interaction — it's punctuated by human interaction." This isn't fully automated — it's human-reviewed. Frame it as: system does the segmentation and pre-population, human approves.
11
"What system does the final order actually get submitted into — your pricing program, directly to the factory, or somewhere else?"
This defines the integration endpoint and determines whether we need an API, a file export, or manual copy-paste from a pre-filled screen. Tom mentioned a "pricing program" — find out what it is specifically.
12
"Are there cases where a room spans two floors, or where a floor release is only partial — not the whole floor at once?"
Edge cases from the Phase Alpha analysis: rooms spanning floors, partial releases, last-minute spec changes. Find these now so the data model handles them. A rigid floor-based model breaks if rooms don't map cleanly to floors.
Step 5 — Follow-Up / CRM Layer
13
"What would a system need to do to make sure no quote slips through — not just the important ones, but all of them?"
Context: Dave gave one contractor 2,000 quotes in a year and got one buy-request back. Tom said the company that called them back on "you weren't low, give me best and final" won the job. The system needs to create that call to action automatically.
14
"If you had one view that showed all open quotes, their dollar value, and which ones need a call this week — would Tom and Dave actually look at it every morning?"
Tests real adoption vs. theoretical interest. Tom said he wants a dashboard for high-value jobs — Marc suggested it on the call and Tom agreed. But Dragonfly was also supposed to be used and wasn't. Probe the adoption question directly.
15
"What's the dollar value on a quote where you'd say — someone absolutely needs to personally call this customer, not just send an email?"
This sets the "high value threshold" for alert triggers. Tom mentioned hospital OR systems and destratification fans as high-value categories — but we need a number. This drives how we configure the follow-up automation.

Platform & API Integration
Goal: Map every connection point between Dragonfly and the outside world — Outlook, Make/Zapier, AlphaTakeoff, QuickBooks, the pricing program. These integrations are what turn Dragonfly from a database into an automation platform.
A1
Does Dragonfly expose a REST API — and if so, what can it do? Create records, update status fields, attach files, trigger workflows?
This is the most important technical question on the call. Everything we want to automate — bid intake creating an opportunity, AlphaTakeoff attaching a takeoff sheet, status changes firing follow-ups — requires API write access. Without it, we're limited to what PhaseAlpha builds natively.
A2
Does Dragonfly support outbound webhooks — so that when a record changes (e.g. status → "Quoted"), it can push a notification to Make or Zapier automatically?
Webhooks are what let us trigger external automations from inside Dragonfly without polling. If Dragonfly fires a webhook when a quote status changes, Make can catch it and immediately send a follow-up email — no manual step required.
A3
Is there a native Outlook / O365 integration — and does it go both ways (emails create records, and records trigger emails)?
Hilberts runs on Outlook. Tom described dragging documents from email into Dragonfly manually. A native O365 integration means ITB emails could auto-create Dragonfly opportunities without Make. Confirm: does this exist, and does it support inbound (email → record) AND outbound (record event → send email)?
A4
Is there a native Make or Zapier connector for Dragonfly — or do we need to use the raw API with custom HTTP actions?
A native Make connector makes building automations dramatically faster — pre-built triggers and actions mean less custom code. If there's no connector, we build against the raw API which is doable but slower and more fragile.
A5
How does AlphaTakeoff output connect to Dragonfly — is it automatic, or does someone manually upload the takeoff spreadsheet to the Dragonfly project record?

This is where the engagement succeeds or fails. Without a defined schema for Job → Floor → Room → Equipment → Order, every process stays manual. The data model question is the most important technical question on this call.

DM3

If adding "Floor" as a new record type breaks the existing project/quote structure, we have a problem. Most flexible platforms support schema evolution — confirm Dragonfly does, and what the process looks like.
DM4

For the room schedule to auto-generate, Room records need: number (CFM, size, static pressure), dropdown (equipment designation), relationship (link to Floor, link to Equipment). If formula fields exist, we can derive values. If not, calculations happen outside Dragonfly.
