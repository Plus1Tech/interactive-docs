System Overview

The HOCK Content Production Agent is a Claude Project that manages the end-to-end production of certification exam prep materials — textbooks, MCQs, slide presentations, narration scripts, and video. Claude generates content; the agent manages approvals, workflow tracking, and external tool actions.

📋
ℹ
This document is the authoritative technical reference for the HOCK Content Production Agent. It covers architecture, workflow logic, connector specs, skill behavior, and maintenance procedures. It is intended for internal use, handoff, and client team reference.
🎯 What the Agent Does

The agent phase-locks all content generation to a strict 8-step production order. It invokes the correct skill at each phase, gates every action behind user approval, tracks workflow state across the session, and pushes finalized artifacts to Monday.com, Dropbox, and Gamma using n8n webhooks and MCP connectors.

System Architecture

How all components connect — the Claude Project, knowledge files, skills, connectors, and external platforms.

🏗
🔁 Data Flow Summary
  • 1
    User → Claude
    User provides prompts, source materials, and approvals. Claude reads system prompt + knowledge files + skills on every session start.
  • 2
    Claude → Content Generation
    Claude invokes the correct skill per phase, applies depth calibration and token optimization, generates content in chat, and pauses for user review at defined checkpoints.
  • 3
    Claude → n8n (POST webhooks)
    On user approval, Claude POSTs to n8n webhook endpoints to save files to Dropbox. n8n handles the Dropbox API calls and returns confirmation.
  • 4
    Claude → Monday (MCP connector)
    Claude uses the Monday MCP connector to update task status, assignee, dates, and attach files to the correct columns on the HOCK board.
  • 5
    Claude → Gamma (MCP connector)
    Claude uses the Gamma MCP connector to create presentations from approved outlines, export PPTX, and add speaker notes. Gamma is asynchronous — Claude waits for completed status before returning the gammaUrl.

Skill Reference

⚡
ℹ
Skills are invoked by their name: frontmatter field, not the filename. If the frontmatter name doesn't match exactly what's in the system prompt, the skill will not fire. Verify frontmatter after every file upload or edit.

depth-calibration-skill

Determines content depth before any generation. Always invoked first.

token-optimization

Sets chunking and batching strategy for confirmed depth. Always second.

textbook-style-guide

Style, structure, and tone standards for all textbook content.

mcq-creation-skill

MCQ construction rules, output format, and quality gates.

slide-creation-skill

Gamma-ready Markdown outline generation rules and Gamma trigger gate.

slide-script-skill

Narration script generation matched to finalized slide deck.

monday-workflow-skill

Monday.com interaction rules — always required Phase 1, user-directed elsewhere.

Maintenance Guide

How to update the project, what to touch when things change, and a changelog for tracking updates over time.

🔧
🔗 When Webhook URLs Change
  • 1
    Update HOCK_Core_Operations.md
    Find the Connector Rules section. Update the Webhook: line for Save File and/or Get File. Do not change the Method or Inputs unless those also changed.
  • 2
    Replace the file in Project Knowledge
    Delete the old version of HOCK_Core_Operations.md from the project. Upload the updated file. Confirm the filename is unchanged.
  • 3
    Update this doc's changelog
    Log the change in the changelog table below with date and description.
📊 When Monday Column Names Change
  • 1
    Update HOCK_Core_Operations.md — Phase Behavior section
    Find the affected phase (e.g., Phase 2 — Textbook Creation). Update every reference to the old column name. Column names must match Monday exactly — Claude uses them verbatim when calling the connector.
  • 2
    Update monday-workflow-SKILL.md
    Find references to the old column name in the Process Steps tables. Update to match.
  • 3
    Replace both files in Project Knowledge
    Upload updated versions. Verify filenames match exactly.
➕ When Adding a New Phase
  • 1
    Update HOCK_System_Prompt.md — Production Order
    Add the new phase to the numbered list. Update the phase numbers of any phases that come after it.
  • 2
    Update HOCK_Core_Operations.md — Workflow State Table + Phase Behavior
    Add a new row to the Workflow State Table. Add a new Phase Behavior section with skills, prerequisites, steps, and Monday fields.
  • 3
    Create or update the skill file if needed
    If the new phase needs a dedicated skill, create the skill file with correct frontmatter. Add it to the Skills Reference table in HOCK_System_Prompt.md.
  • 4
    Upload all changed files to Project Knowledge
    Replace system prompt in Project Instructions field. Replace knowledge files. Upload new skill file.
🔄 When Updating a Skill File
  • 1
    Never change the frontmatter name field
    The name: in the frontmatter must match exactly what the system prompt references. Changing it breaks skill invocation silently — Claude won't error, it just won't apply the skill.
  • 2
    Delete the old version before uploading the new one
    Claude Projects don't version-control files. If you upload without deleting, both versions may exist and cause conflicts.
  • 3
    Test the updated skill with a verification prompt
    Open a new conversation in the project. Trigger the phase that uses the updated skill and confirm it behaves as expected before using in production.
